# xingshu 与 xingshu-v2 合并设计

## 背景

本次工作将 `1017293270/xingshu-v2` 的新增提交整合到原仓库
`1017293270/xingshu`，并继续以本地 `E:\starmath`、远端
`xingshu.git` 作为唯一主仓库。

只读审计确认两个仓库来自同一条提交链：原仓库 `master` 位于
`ab23765`，v2 的 `master` 位于 `4d91615`，原仓库 HEAD 是 v2 HEAD
的直接祖先。v2 仅新增以下两个提交：

- `f031c62 redesign: 我的看板三档响应式重构`
- `4d91615 test: 视觉测试播种登录态并对齐当前实现`

两个仓库的目录、依赖、锁文件和项目约束一致。新增内容只涉及 8 个
文件，集中在“我的看板”响应式布局、ECharts 自适应和 Playwright
视觉测试。

本设计文档提交追加在本地 `E:\xingshu-v2` 的 `4d91615` 之后，作为
合并暂存链的一部分；它不会推送到 `xingshu-v2.git`，而会随同前述
两个提交一起快进到原仓库。

## 目标

- 保留 v2 两个提交的原始提交身份和顺序。
- 将原仓库 `master` 以纯快进方式推进到 v2 的最新线性提交。
- 不产生合并提交，不重写既有历史，不覆盖未提交改动。
- 合并后完成单元测试、生产构建和多尺寸视觉测试。
- 验证通过后将结果推送到 `xingshu.git` 的 `master`。

## 非目标

- 不改动产品功能或视觉实现。
- 不修改依赖版本、锁文件或远端仓库配置。
- 不删除本地 `E:\xingshu-v2` 克隆，也不向 `xingshu-v2.git` 推送。
- 不使用 squash、rebase 或 cherry-pick 重建已有提交。

## 方案选择

采用快进合并。与 cherry-pick 相比，它不会生成新的提交身份；与
squash 或文件复制相比，它保留完整的重构与测试历史。由于
`ab23765` 是 v2 HEAD 的祖先，`git merge --ff-only` 能在祖先关系
变化或出现分叉时直接失败，而不会自动创建意外的合并提交。

## 执行流程

1. 确认 `E:\starmath` 和 `E:\xingshu-v2` 的工作区均干净，并获取
   两个远端的最新引用；确认 v2 本地分支只比其远端多出已批准的规范
   文档提交。
2. 再次校验原仓库 HEAD 是 `E:\xingshu-v2` 本地 HEAD 的祖先，并
   确认差异只包含 v2 的两个新增提交和已批准的规范文档提交。
3. 在 `E:\starmath` 中从本地 `E:\xingshu-v2` 获取临时引用，执行
   仅允许快进的合并；不新增或改写永久远端配置。
4. 确认原仓库工作区干净、提交链连续，且 `origin` 仍指向
   `https://github.com/1017293270/xingshu.git`。
5. 运行验证；全部通过后，将 `master` 快进推送到原仓库远端。
6. 保留 `E:\xingshu-v2` 作为本地来源副本，不删除目录，不修改其
   远端。

## 验证

- `npm test`：运行 Vitest 单元与组件测试。
- `npm run build`：运行 TypeScript 编译和 Vite 生产构建。
- `npm run test:visual`：运行 Playwright 视觉测试，覆盖 1440×900、
  1672×941、2200×944 和 390×844。
- 检查 Playwright 生成的“我的看板”截图，确认卡片网格、图表尺寸、
  文本边界、侧栏和页面横向溢出符合现有规范。

## 失败处理与回退

- 任一工作区不干净时停止，不自动暂存、覆盖或丢弃用户改动。
- 祖先关系校验失败或 `--ff-only` 拒绝合并时停止，不改用普通 merge。
- 测试、构建或视觉检查失败时不推送；先报告失败证据并定位原因。
- 推送前记录原远端 HEAD `ab23765`。若尚未推送，本地可把主分支恢复
  到该提交；若已经推送且需要回退，使用显式 revert，不强推重写历史。

## 完成标准

- 原仓库包含 v2 的新增提交和本设计文档提交，历史保持线性。
- 所有自动化验证通过，视觉截图完成检查。
- `xingshu.git/master` 与本地 `E:\starmath/master` 指向同一提交。
- 两个本地工作区均无未提交改动。
