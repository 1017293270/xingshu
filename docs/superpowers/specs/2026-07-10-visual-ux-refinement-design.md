# 星数全站视觉与体验精修设计

## 1. 目标

在不改变星数“可信数据智能 / 企业智能中枢”定位、不新增无关业务模块、
不偏离现有参考图的前提下，系统性提升全站视觉一致性、核心流程可用性、
移动端体验、状态反馈、可访问性和轻量动效质量。

完成后的产品应具备以下特征：

- 桌面端保持浅冰蓝底、白色侧栏、蓝色品牌识别和深蓝信息层级。
- 移动端不再是桌面布局的简单纵向堆叠，所有核心模块均可导航、可操作。
- 问数、登录、历史、上传、同步、生成、配置、看板编辑等关键流程有真实、
  明确且不误导的 loading / empty / success / error 状态。
- 视觉 token、图标、圆角、阴影、排版、焦点、控件高度和动效时长形成统一系统。
- 动效克制、短促、可中断，并完整支持 `prefers-reduced-motion`。
- 所有改动继续遵守现有前端服务边界，不在页面内接入临时后端。

## 3. 已有优势

- 主色、背景、白卡、细边框和低阴影已形成稳定的企业产品气质。
- 桌面侧栏结构、页面标题、主要 CTA 与数据卡已有较高一致性。
- 数据资产看板、数据资产管理和云盘已具备成熟的信息密度与网格。
- 登录流程已有 label、required、loading、取消和后端错误展示。
- 固定业务看板已关闭大规模 ECharts 入场动画，符合企业工作台的稳定感。
- 现有视觉测试覆盖 1440、1672、2200 和 390 宽度，并检查横向溢出。

## 4. 核心问题

### P0：核心任务阻断或明显误导

1. 900px 以下直接隐藏主导航和账户入口，用户无法进入绝大多数模块。
2. 移动问数页输入区位于首屏之外，页面中央没有任务提示或快捷问题。
3. 移动登录先展示整套品牌能力，登录表单约一屏后才出现。
4. 看板编辑器 iframe 连接失败时仍显示“已嵌入”，状态自相矛盾。
5. 通用命令框缺少正式规范要求的附件入口。

### P1：系统性体验问题

1. CSS token 与 TypeScript token 双份维护；token 缺少间距、排版、焦点、控件、
   z-index 和 motion 维度。
2. `pages.css` 集中承载所有页面样式，跨页面覆盖和失效样式持续累积。
3. 首页彩色渐变 PNG、Phosphor、欢迎/登录 Image2、历史 SVG 等至少五套图标
   语言并存；首页图标命中“糖果渐变 / 玩具化”禁止项。
4. 辅助文字和部分蓝色小链接对比不足，多个自定义输入与按钮没有统一焦点环。
5. 多数 React Query 页面缺少 loading / error / retry / empty；历史页加载时会短暂
   显示假空态。
6. 多处操作仅显示“成功”状态条但不改变内容，包括云盘上传、知识库同步、写作
   附件、看板切换、处理预警和资产类型切换。
7. `XsEChart` 在 option 对象变化时销毁实例；生成图表 option 每次 render 新建，
   存在闪烁、重复初始化和默认动画重播风险。
8. 图表缺少同源文本摘要或可展开数据表；部分表格缺 caption / header 语义。
9. 路由级 Suspense 会连共享外壳一起替换，fallback 是无样式裸文本。
10. 页面和侧栏并存滚动条；欢迎、登录等页面存在双重纵向滚动风险。

### P2：精致度与信息质量问题

- 数据资产日期、文稿日期等仍为 2024 年，降低“可信数据”感受。
- 桌面首页 7 列卡片在中间宽度拥挤；移动端 KPI 没有 2×2 紧凑布局。
- `查看 / 查看明细 / 忘记密码` 等动作目标过小、颜色过淡。
- 云盘出现“后续真实云盘服务只需替换数据适配层”等面向开发者的文案。
- 数据卡图标复杂度、细节密度和底板形式不统一。
- 生成、同步、加载等状态存在多个 spinner / dots / shimmer 同时运行的问题。

## 5. 方案比较

### 方案 A：基础优先的系统化精修（采用）

先修移动导航、登录/问数/编辑器 P0，再统一 token、焦点、异步状态、图表生命周期、
图标与 motion，最后逐页做密度和文案精修。

优点：同时解决可用性和视觉根因，后续页面不会继续漂移；适合正式产品演进。
代价：涉及共享外壳、基础组件和多个页面，需要分阶段实施和完整回归。

### 方案 B：只做表层视觉润色（不采用）

仅调间距、阴影、颜色、圆角和 hover。

优点：变更小、视觉上短期见效。缺点：移动导航、假动作、错误态、可访问性和图表
闪烁等核心问题仍然存在，无法达到“用户体验更好”的目标。

### 方案 C：重新生成整套视觉资产并大幅改版（不采用）

通过 Image2 重做首页图标、插画和各页装饰。

优点：视觉差异明显。缺点：违反项目优先复用真实图标库和现有参考图的约束，容易
变成营销页、糖果渐变或玩具化，也无法解决交互结构问题。

## 6. 设计方案

### 6.1 设计系统基础

- 以 `xingshuTokens.ts` 为品牌 token 的唯一源，CSS variables 与 Ant Design 主题
  必须能被一致性测试验证。
- 增加 spacing、font size、line height、weight、control height、focus ring、
  z-index、motion duration 和 easing token。
- 辅助文字改为在白底与浅冰蓝底均满足普通小字对比要求的深度；主蓝小文本使用
  深主蓝，品牌主蓝继续用于大面积控件和图形。
- 统一 `:focus-visible`：3px 半透明品牌蓝外环，不通过位移表达焦点。
- 拆分 CSS：共享 primitive / pattern 保留在 `components/xs`，页面样式回到对应 feature。
  第一轮只拆与改动范围直接相关的规则，避免一次性机械重写全部 2000+ 行 CSS。
- 高频模式封装为 `XsPageHeader`、`XsAsyncPanel`、`XsEmptyState`、`XsFilterBar`、
  `XsDataCard`、`XsIconTile` 和 `XsChartCard`。

### 6.2 外壳与移动导航

- 将 `XsShell` 提升为持久布局，路由只替换主内容，侧栏不再随页面 remount。
- 桌面保留 320px 白侧栏；仅在内容真实溢出时显示细滚动条。
- 移动端使用约 64px 的紧凑顶栏：小 Logo、当前页标题、菜单按钮、账户入口。
- 菜单按钮打开 Ant Design Drawer，复用桌面导航数据，包含全部现有功能和账户操作。
- Drawer 使用 `aria-expanded`、焦点陷阱和关闭后焦点回归；不新增底栏或新业务入口。
- 路由切换重置主内容滚动位置，更新文档标题，并把焦点移动到页面主标题。

### 6.3 问数与命令输入

- 空白问数工作区增加克制的品牌空态：标题、说明、3–4 个快捷问题和数据权限提示。
  优先复用现有品牌/图标资产；只有现有资产无法表达时才生成一张专用空态插画。
- 桌面输入框保持底部对齐；移动端固定在安全区上方，首屏即可操作。
- 恢复附件按钮并接入隐藏 file input / 本地上传队列 UI；真实上传仍经过 `services/`。
- 语音建立 idle → permission → recording → processing → success/error 状态，不再只显示
  一条“已准备”提示；支持取消并提供 `aria-pressed`。
- 空输入禁用发送；支持 Ctrl/Cmd+Enter；发送中提供 Stop，旧流必须通过 runId/turnId
  与当前 turn 绑定，新建对话先 abort。
- 每个作用域只保留一个主运动信号：Progress + 阶段文案，移除重复 spinner/shimmer。

### 6.4 登录与欢迎页

- 移动端把登录卡前置到品牌标题之后，能力说明缩为 2–3 个信任标签并后置。
- 桌面继续保留双栏结构，但允许矮屏纵向滚动，不使用会裁切内容的固定 overflow。
- 修复双滚动；顶部 Logo 与安全标识在移动端缩小并避免换行拥挤。
- 登录成功读取并校验 `location.state.from` 后回到原任务；认证和空间初始化全部成功后
  才提交登录态，失败时清理半完成会话。
- 欢迎页保留品牌插画，不扩展为营销 landing page；提升插画清晰度和次级文本对比。

### 6.5 异步状态与真实反馈

- `XsAsyncPanel` 统一 pending、refreshing、empty、error、retry 和 content 状态。
- 首次 pending 延迟 180–250ms 再显示 skeleton，避免短请求闪烁；refreshing 保留旧数据。
- History 不得在 fetching 时显示“暂无历史”。空态需提供“开始新对话”主动作。
- 看板编辑器通过 iframe `onLoad`、超时与错误探测显示 loading / ready / error；连接失败时
  不得显示“已嵌入”，提供重试和配置说明。
- 暂未实现的操作必须明确 disabled + “即将开放”，或实现真实可见的本地状态变化；禁止
  仅用成功提示假装完成。

### 6.6 图表与数据体验

- ECharts 实例只初始化一次；option 变化调用 `setOption`，unmount 时才 dispose。
- ResizeObserver 回调通过 requestAnimationFrame 合并；避免同时给每个图表注册冗余 resize。
- 固定业务看板继续使用 `animation:false`；数据切换可使用 220–280ms 更新动画。
- AI 生成图表根据 reduced-motion 决定是否动画，ready 后最多 160ms opacity 过渡。
- 每张核心图表增加简短趋势摘要和可展开数据表，表格提供 caption / th scope。
- 移动看板先呈现指标结论；复杂图表提供横向查看或全屏入口，不简单压缩全部标签。

### 6.7 图标与资产

- 常规导航和应用卡统一使用 Phosphor 线性图标，通过 `XsIconTile` 的底板、颜色和青色
  节点形成“星轨 Agent Icons”，不为每个图标叠加轨道纹理。
- 首页移除彩色渐变方形 App PNG 和挥手 emoji，恢复企业级线性图标语言。
- 登录和品牌级插画可继续使用已有 Image2 资产；不生成新人物、设备、海报或营销场景。
- 建立 `iconRegistry.ts`，记录语义、来源、使用尺寸和页面，清理完全重复资产。
- 原始 sheet/source 移出运行时 `src/assets`；展示资产按实际 2x 尺寸导出并转 WebP/压缩。
- 首页 1MB 波浪背景转为视觉无损的 WebP，并保留相同构图与透明度。

### 6.8 动效规范

新增 motion token：

- fast：120ms；base：180ms；slow：260ms。
- 标准缓动：`cubic-bezier(.2, 0, 0, 1)`。
- 最大常规位移：4px。

使用范围：

- 按钮 / 导航：颜色、边框、阴影 140–160ms；pressed 80ms `scale(.985)`。
- 可点击卡片：hover `translateY(-2px)` + 低阴影 160ms，静态数据卡不漂浮。
- 页面内容：首次进入 180ms opacity + `translateY(4px)`；不做退出动画。
- Drawer / disclosure：caret 旋转和内容淡入 180ms，不用脆弱的 JS 高度动画。
- 状态条：160ms opacity + 4px；固定状态槽避免插入时布局跳。
- 图表更新：240ms；密集看板首次加载不同时播放 6–10 个生长动画。

明确禁止：持续背景漂浮、全页视差、弹簧 bounce、数字滚轮、长列表大规模 stagger、
逐 token 平滑滚动、每张卡片星轨粒子。reduced-motion 下关闭非必要位移、无限动画和
ECharts 动画，同时保留静态颜色、图标和文字反馈。

### 6.9 逐页精修

- 首页：中宽屏改为容器查询下的 3/4 列；保持参考图的桌面标题型卡片密度，
  通过完整 accessible name 和移动端说明补足语义；移除玩具化图标。
- 历史：筛选与空态使用完整内容轨道，空态居中并带 CTA。
- 制表：修复标题提前截断；重复“复制”动作在 hover/focus 时强化但保持键盘可见。
- 写作：压缩首屏输入区高度，场景卡和文稿列表更早进入视口；附件显示队列。
- 云盘：删除内部实现文案，补“GB / 个空间”等单位，上传/同步显示进度和结果。
- AI 配置：提供 sticky 操作栏，测试连接结果使用 success/error tone，不统一显示蓝色。
- 数据资产：修正 freshness 文案；移动 KPI 2×2；标签支持横向滚动和边缘渐隐提示。

## 7. 验证与完成标准

### 自动化

- Vitest / React Testing Library：基础组件、移动导航、附件、流取消、异步状态、登录回跳、
  ECharts 生命周期、reduced-motion。
- Playwright：1440、1672、2200、390；移动菜单、键盘焦点、错误/空/加载状态、无横向溢出。
- 构建：TypeScript 和 Vite 生产构建通过。
- 可访问性：引入 axe 或等价自动检查，并保留人工键盘、焦点和读屏语义检查。

### 视觉验收

- 每个改造页面都与现有参考图和改造前截图进行同视口并排比较。
- 核心页至少覆盖 desktop 1440/1672/2200 与 mobile 390。
- 不出现双滚动条、无意义首屏空白、被裁切 tab、过淡小链接或不可见焦点。
- 不出现深色、霓虹、玩具化、3D 糖果图标、营销海报或新增无关模块。

### 功能验收

- 移动端可到达全部现有路由和账户操作。
- 登录、问数、上传、同步、生成、看板编辑的状态与实际结果一致。
- 暂不可用功能明确标识，不以成功提示伪装完成。
- `prefers-reduced-motion` 下核心流程仍有完整静态反馈。

## 8. 实施顺序

1. 设计 token、焦点、motion、持久外壳和移动导航。
2. 登录移动任务顺序、问数空态/输入区/流生命周期、编辑器错误态。
3. 统一异步状态、图表生命周期与图表数据替代。
4. 首页图标、附件/语音、逐页密度、文案和移动数据布局。
5. 资产压缩、CSS 聚焦拆分、全站视觉/键盘/性能回归。

该顺序解决 P0 后再铺开系统性精修，不把“更精致”误解成仅增加装饰。
